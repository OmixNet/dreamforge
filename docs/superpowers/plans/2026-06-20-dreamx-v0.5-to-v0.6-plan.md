# DreamX v0.5 to v0.6 Implementation Plan

> **For implementer:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task.

## Goal

Ship DreamX v0.5 only after a real OpenRouter end-to-end run succeeds, then use v0.6 to turn the current provider plumbing into a stable multi-provider LLM layer.

Current status:

- DreamX UI and DreamVault CLI bridge are healthy.
- OpenRouter provider settings and Keychain plumbing are implemented.
- DreamVault has an OpenAI-compatible provider implementation.
- PR 30 routes configured providers through `dream run --llm openai`, so DreamVault reaches the `.openaiCompat` path.
- The latest GUI run was not a real cloud-LLM E2E: DreamPanel reported no work and `budget 0 call(s)`.
- PR 31 cleans visible fixture residue and adds an output check that rejects idle/no-op runs.

## Non-Negotiable Invariants

- API key values must never be stored in `settings.json`, localStorage, exported settings, logs, test fixtures, or git.
- Keychain owns provider API key values.
- DreamX may store provider metadata and env-var names, but not secrets.
- `com.biomatrix.dreamforge` bundle identifier and config namespace stay stable until a dedicated namespace migration.
- Raw files remain immutable user records; engine output goes through controlled notes/wiki/MEMORY flows.
- v0.5 is not shipped until one real provider call is observed.

## Phase 3 - v0.6 Multi-Provider Architecture

**Objective:** Expand from OpenRouter/OpenAI-compatible to a deliberate provider system.

Recommended order:

1. OpenAI-compatible providers as one adapter family.
2. Anthropic native adapter.
3. Gemini native adapter.
4. Optional OpenAI direct provider.

Tasks:

- [ ] Define provider capabilities: chat, model list, timeout, base URL, required secret name.
- [ ] Keep OpenRouter as the first production provider.
- [ ] Add Anthropic only after OpenRouter E2E is stable.
- [ ] Add Gemini after Anthropic, because Gemini request/response shape differs more.
- [ ] Keep provider-specific parsing in DreamVault Swift, not DreamX UI.
- [ ] Keep provider selection and secret management in DreamX Settings.

Verification:

- [ ] Each provider has response parsing tests.
- [ ] Each provider has missing-key and auth-error tests.
- [ ] At least OpenRouter has a real manual E2E before v0.6 tag.
