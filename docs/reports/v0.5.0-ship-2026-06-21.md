## Closed-loop data flow (verified end-to-end)

```
DreamX Settings → AI → OpenRouter/SiliconFlow
    └─→ saveAiModelProviderApiKey (Keychain, PR 25)
          └─→ writeActiveLlmApiKeyEnv + writeActiveLlmApiKeyProviderId (localStorage, PR 26/27)
                └─→ DreamPanel reads both on "Run Dream" click (PR 27)
                      └─→ dreamvault_run { llmApiKeyProviderId, llmApiKeyEnv, ... } (PR 27)
                            ├─→ PR 30: --llm openai flag added (closed-loop 4th fix)
                            └─→ inject_api_key (PR 27):
                                  ├─→ keychain_read_value (Keychain, PR 27)
                                  └─→ Command::env("DREAMFORGE_LLM_API_KEY", value)
                                        └─→ dream subprocess env
                                              └─→ OpenAICompatibleProvider.init (DreamVault PR 28)
                                                    └─→ URLSession POST /v1/chat/completions
                                                          └─→ OpenRouter/SiliconFlow response
                                                                └─→ Consolidator → Persister → MEMORY.md
```

**Locked invariants (test-verified)**:
- key value 不进 localStorage / settings.json / settings export / log / panic message
- service = `com.biomatrix.dreamforge.ai-provider`, account = validated provider id
- has_* returns `{ provider_id, configured: bool }` only — value fetched and dropped internally
- errSecItemNotFound → `Ok(false)` (UI shows "API key: not set" without error)
- delete on missing → `Ok(())` (idempotent)

## What this enables

- A user can configure any OpenAI-compatible cloud provider in DreamX Settings → AI, save the API key (stored in macOS Keychain only), select model, click Run Dream. The end-to-end path (Settings → Keychain → Rust inject → dream CLI → Swift provider → cloud API → Consolidator) is fully wired and verified.
- Provider-specific protocol differences (Anthropic Messages API, Gemini generateContent) are deferred to v0.6. v0.5's `OpenAICompatibleProvider` covers OpenRouter, SiliconFlow, OpenAI direct, and any other `sk-...` or `sk-or-v1-...` provider.
- Key material never appears in `settings.json`, localStorage, exported settings envelope, or logs. The Keychain is the only place a provider key value lives, and only the running `dream` subprocess sees it (transient env var).

## Scope discipline record (user enforced)

- **NO** Anthropic / Gemini adapter (v0.6) — different protocols, separate PR
- **NO** streaming response (Consolidator expects single string completion)
- **NO** HTTP smoke test in Settings UI (P2b scope: no real OS / network in unit tests)
- **NO** multi-provider adapter bundled into single PR (one provider per PR; OpenAI-compat covers most)

**Closed-loop lessons learned (write-down)**:
- "Security boundary test ≠ end-to-end test" (PR 24/25/26 各自测试 pass 但组合不能 end-to-end) → trace 跨 PR data flow 在 next-PR review 阶段,不能只看 PR 自身
- "Trace env 边界" (Keychain → env var → dream subprocess 是跨 PR 的多层数据流)
- 第一 PR (e.g. PR 24 的 "env var wiring") 涉及 multi-layer data flow 时,commit message 要 explicit 写 "next PR 必须 trace 到 key 真实存储位置"
