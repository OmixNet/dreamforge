## What landed in the PR 52-54 series (this ship doc's scope)

### PR 52 (commits `8d18c44` + `4a55eb6`) — DreamPanel consumes typed vault stats

PR 50c.2 (v0.6.0 addendum) made the **Editor empty state** show "5 candidates · 7 processed · 1 archived" when typed stats are available. The DreamPanel (right column) was the missing piece — users would see the detailed numbers in the empty editor but the right-column panel still showed the PR 47 simple format.

**Phase A (`8d18c44`)**: DreamPanel fetches typed stats via `dreamvault_status_json`, mirrors the Editor's display logic (3 numbers required for detailed format; otherwise simple fallback). New `useEffect` to refresh after `Run Dream` completes (so the just-updated numbers show up without a manual click).

**Phase B (`4a55eb6`, user self-fix)**: typed section hidden on first view (skip-first-mount guard). The 16 existing `mockResolvedValueOnce` tests in DreamPanel.test use command-agnostic mocks that would be consumed by a 2nd `invoke` on mount. Clicking the "Status" button in the panel populates the section. UX trade-off: 0-click → 1-click, but all existing tests stay green without mass-mock refactor.

3 new i18n keys (`dreamPanel.stats.lastDream` / `lastReport`), 4 new tests, +22 dreamforge tests.

### PR 53 series (commits `bcda17e` / `2ef25a8` / `e5ce610` / `d04886e` / `ce5a2d5` / `e069b64`) — Run Dream state experience

Before PR 53, clicking "Run Dream" produced a flat sequence: spinner → log dump → "Done." Users couldn't tell from the UI whether the run actually did work, succeeded with no new memory, or failed.

**Phase 1 (`bcda17e`)** — `src/lib/dreamRunSummary.ts` pure parser. `parseDreamRunSummary(stdout)` returns `{kind: 'completed' | 'noop' | 'unknown', rawCollected, integrated, reportPath}`. Pure helper, no IO. Recognizes Chinese labels (`收集 raw` / `通过整合` / `无需事项（无新 raw, 无矛盾需解决）`) AND English (`collected raw` / `integrated`).

**Phase 2 (`2ef25a8`)** — 4-state UI in DreamPanel: `Running` (spinner) / `Completed` (green check + counts) / `No new work` (info, no raw/矛盾 to consolidate) / `Failed` (reuses PR 36 `ProviderErrorView`, no double-state). State machine driven by parser output.

**Phase 3 (`e5ce610`)** — "Open latest report" action. 2-step fallback: `vaultBridge.openReport(reportPath)` if available, else raw `open_path` Tauri command. Disabled when no report path or no Tauri context.

**Phase 4 (`d04886e`)** — 20-locale i18n for the 4 states. English placeholders for non-en locales (same pattern PR 54 used).

**Self-fix (`ce5a2d5`, user)** — Real dream CLI emits a Chinese no-work line: `dream: 一无事事（无新 raw，无矛盾需要裁决）` (PR 50 series + PR 51a fixture confirmed this). PR 53's parser only matched `无需事项`, so it classified real output as `unknown`. New `isExplicitNoWorkOutput()` helper with regex `/无需事项|一无事事|无新\s*raw|no\s+new\s+raw|nothing\s+to\s+do/i` covers both variants. **Lesson locked**: when a parser is built on synthetic data, the first real-data run exposes patterns the parser missed. Always add a real-data fixture and a self-fix PR.

**Plan doc (`e069b64`)** — `docs/superpowers/plans/2026-06-25-pr53-run-dream-status-experience.md` for archeology.

5 parser tests + 4 UI tests, +9 dreamforge tests, +20-locale i18n parity test still 15/15.

### PR 54 series (commits `6bc6d69` / `113603f` / `3116cf6` / `552a979` / `c3469aa` / `6e1cea5` / `46f8e21`) — Settings AI closure

The Settings → AI panel had 4 small UX gaps flagged in the user's "v0.6.x polish" backlog. PR 54 is 4 sub-tasks, each a 1-commit closeout, then 2 follow-ups:

**Task 1 (`6bc6d69`)** — **Active banner** (`ActiveProviderBanner` component). Replaces the easy-to-miss "Active" badge + muted summary text. Primary border + bg, ✓ icon, provider name + model id, Clear button. data-testid `ai-providers-active-banner`. 2 new tests.

**Task 2 (`113603f`)** — **Base URL /v1 hint** (`BaseUrlV1Hint` component). Inline under the Base URL input. Subtle hint when URL is bare domain (`https://api.openai.com`); prominent warning when URL ends with `/v1` (`https://api.openai.com/v1` — would double-insert). Only shown for `open_ai` / `open_router` / `open_ai_compatible` kinds (Anthropic uses `/v1/messages` differently, Gemini uses `/v1beta/models/...:generateContent` — `/v1` rule does not apply). 2 new tests + 1 skipped (Radix Select doesn't accept `fireEvent.change`).

**Task 3 (`3116cf6`)** — **Keychain status dot** (`KeychainStatusDot` component). Colored dot for at-a-glance state recognition. 4 states mapped to colors: green (local_file + configured), red (local_file + !configured), amber (env + env_var set), red (env without var or none). ProviderList row renders the dot inline next to the base URL. data-testid `ai-providers-keychain-dot-{provider.id}` + `data-state` attribute. **Mock shape fix**: `has_ai_model_provider_api_key` returns `{provider_id, configured: bool}`, not bare bool. The previous test mocks returned plain booleans which made `result.configured` undefined → dot always rendered 'missing'. 3 new tests.

**Task 4 (`552a979`)** — **Provider delete confirm dialog** (`DeleteProviderDialog` component, shadcn Dialog). Replaces the single-click Remove → delete flow, which was too easy to misfire. Title with `{name}` placeholder, body explains side effects (Keychain removal, no vault file deletion), Cancel + Delete buttons. `pendingDelete` state holds the provider awaiting confirmation. ProviderList prop API change: `onRemove(providerId)` → `onRequestDelete(provider)`. The actual delete is gated on dialog confirmation. Form draft survives cancel. 4 new tests + 2 PR 43 tests updated to click through the dialog (Remove → Delete) instead of Remove alone. 4 new i18n keys × 20 locales.

**Plan doc (`c3469aa`)** — `docs/superpowers/plans/2026-06-26-pr54-settings-ai-closure.md` for archeology.

### PR 54 follow-ups (LaunchServices + Keychain IPC)

**`6e1cea5` (PR 54.6)** — LaunchServices stale-registration cleanup. macOS LaunchServices keeps registrations for every .app bundle ever opened/scanned, even after the .app is moved to Trash. The DreamForge → DreamX rebrand (PR 17-19 era) left a stale registration at `~/.Trash/DreamForge.app`. When `open` resolved `com.biomatrix.dreamforge`, it hit the stale entry first, couldn't find the executable at the trashed path, and reported `kLSNoExecutableErr: The executable is missing`. The .app bundle itself was fine (executable present, codesign clean, bundle ID correct) — pure environment state. Fix: `lsregister -u <stale-path>` for 5 stale entries (1 rebrand leftover + 3 v0.6 rebuilds + 1 in `/private/tmp/DreamX.app`). After cleanup, `open DreamX.app` succeeded. Docs-only commit (157 lines) — no source code change. The doc captures symptom + root cause + diagnose + fix + nuclear option + prevention pattern. **Lesson locked in agent memory `macos-toolchain-gotchas.md` §5** — reusable across any macOS app project, not just dreamforge.

**`46f8e21` (PR 54.7, user self-fix)** — Keychain IPC native fallback. The previous `aiProviderSecrets.ts` had:
```ts
if (!isTauri()) return  // ← bug
await invoke('save_ai_model_provider_api_key', ...)
```
The early `return` meant: in any code path where `isTauri()` returns `false` (browser dev mode, edge cases in Tauri detection), the save/delete IPC was skipped, falling through to mock handlers. In the native .app, this manifested as "I clicked Save API key, but Rust dreamvault_run didn't see it on the next Run Dream." Fix: new `keychainInvokeIfAvailable()` helper that **tries invoke first, falls back to mock on error**. This is the right call — even if `isTauri()` reports false, the `invoke` may still be wired up; if it throws, we're in browser mode and mock is the only path. Also renamed `providerId` → `provider_id` in `ProviderKeyStatus` to match the Rust return shape (the TS type was wrong — it claimed `providerId` but Rust actually returns `provider_id`, so the existing type didn't match reality). 4 new tests in `src/utils/aiProviderSecrets.test.ts` cover the 4 paths. **Lesson locked**: "isTauri() ? invoke : mockInvoke" is the right dispatch — but `isTauri()` can lie. The correct pattern is "try invoke, fall back to mock on error" rather than "trust the detection, skip the call." This is a stronger version of the PR 50c.1 lesson (`isTauri() ? invoke : mockInvoke` is necessary but not sufficient — also need to handle invoke failure gracefully).

## Post-ship addendum — PR 54.6 + PR 54.7 lessons

After the PR 54 main series (54.1-54.5) shipped to origin/main, two follow-ups were necessary:

### PR 54.6 — LaunchServices stale registration (env-state only)

- **Symptom**: `open DreamX.app` fails with `kLSNoExecutableErr: The executable is missing`. The .app bundle is fine — `Contents/MacOS/dreamforge` is present, executable, codesign-clean.
- **Root cause**: macOS LaunchServices kept 5 stale registrations for `.app` bundles with bundle ID `com.biomatrix.dreamforge`:
  - `/Users/biomatrix/.Trash/DreamForge.app` (PR 17-19 rebrand leftover)
  - `/Users/biomatrix/.Trash/DreamX.app` (v0.6.0 release)
  - `/Users/biomatrix/.Trash/DreamX 06.53.02.app` (timestamped build)
  - `/Users/biomatrix/.Trash/DreamX 07.04.25.app` (timestamped build)
  - `/private/tmp/DreamX.app` (build copied to /tmp for some test)
- **Fix**: `lsregister -u <stale-path>` for each non-current entry, then `mavis-trash` the files.
- **Lesson**: bundle ID is the LaunchServices lookup key. Renaming the .app doesn't invalidate the old registration. Every `pnpm tauri build` adds a new entry; if a previous .app is moved to Trash, the stale entry survives unless explicitly unregistered. Pattern: `lsregister -u <old.app>` **before** `mavis-trash` the file.
- **Memory**: locked in `macos-toolchain-gotchas.md` §5. Reusable across any macOS app project (Tauri / Electron / Xcode / Swift CLI / anything bundled as `.app`).

### PR 54.7 — Keychain IPC native fallback (user self-fix)

- **Symptom**: "Save API key" worked in the form, but Run Dream didn't see the key. The Keychain entry was there; the env var wiring was right; the dream CLI subprocess got nothing.
- **Root cause**: `aiProviderSecrets.ts` had `if (!isTauri()) return` at the top of `saveAiModelProviderApiKey` / `deleteAiModelProviderApiKey`. In any code path where `isTauri()` reports `false` (browser dev mode, edge cases in detection), the early return skipped the IPC entirely. The key was never written to Keychain in those paths.
- **Fix**: new `keychainInvokeIfAvailable()` helper — `try invoke; if it throws AND we're not in Tauri, fall back to mock`. Rename `providerId` → `provider_id` in `ProviderKeyStatus` to match Rust's actual return shape.
- **Lesson**: "isTauri() ? invoke : mockInvoke" (PR 50c.1 pattern) is necessary but not sufficient. `isTauri()` can lie (returns false even when invoke is wired up). The correct pattern is "try invoke, fall back to mock on error" rather than "trust the detection, skip the call." This is a stronger version of PR 50c.1's lesson.
- **Verification**: 4 new tests in `aiProviderSecrets.test.ts` cover the 4 paths (save via native / delete via native / has-key native priority / fallback when invoke throws).
