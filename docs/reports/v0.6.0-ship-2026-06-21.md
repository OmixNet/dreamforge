## Status: API-complete and test-verified

v0.6.0 ships the **adapter capability** for Anthropic Messages API and Google Gemini generateContent API. The full data flow is wired and unit-test-verified, including:

- Per-provider request shape (Anthropic `x-api-key` + `anthropic-version`, Gemini `x-goog-api-key` + literal `:generateContent` suffix)
- 6-category error contract per provider (`[ANTHROPIC_*]` and `[GEMINI_*]` tag prefixes, parallel to PR 34's `[OPENAI_*]`)
- DreamX UI parser recognizing all 18 tags (6 categories × 3 providers) → same short actionable copy regardless of provider
- Strict-concurrency clean on all new code
- `swift build -c release` clean on both new providers
- Dream CLI `--llm anthropic` / `--llm gemini` doesn't crash on idle path

## What this does NOT include (honest scope)

**Real Anthropic / Gemini end-to-end with user-provided keys is NOT verified for v0.6.0 ship.** v0.5.0's PR 32 ship gate was a real E2E with SiliconFlow + DeepSeek-V4-Flash (the user provided a working key, the dream CLI actually called the cloud API, the gate script confirmed a real call happened). The v0.6.0 ship gate is **deliberately deferred** because:

- Mavis does not have Anthropic / Gemini API keys (no way to verify real auth headers or response parsing without hitting the real endpoints)
- The user has not provided real keys for these providers yet
- The provider contracts are well-defined public APIs (Anthropic Messages, Gemini generateContent) and the unit tests cover the full request/response shape including all 6 error categories
- Forcing a real E2E here would block v0.6.0 ship on a procurement step (key acquisition) rather than engineering verification

The wiring is provider-correct by construction (URL shape, headers, body, response parser all match the documented public APIs) and the existing v0.5.0 E2E proves the closed-loop data flow (Keychain → env var → dream subprocess → Swift provider → HTTP) works end-to-end for one cloud provider. The new providers share the same data flow.

**Real E2E verification will land in v0.6.x** as soon as the user provides keys. See "v0.6.x backlog" below.

## What landed

### PR 34 — Provider Error UX (commits `b0c3f73` Swift + `5e8cb57` dreamforge)

The dream CLI's stderr is the only signal DreamX has to surface LLM provider failures. v0.5 PR 28 had 4 error categories which DreamX UI couldn't distinguish well — auth failures and server crashes looked the same in the UI, and the user saw the raw stderr including the `[OPENAI_*]` tag prefix and the body (which may contain the API key value if a misconfigured server echoed it back).

**Phase A (DreamVault Swift, `b0c3f73`)**: `OpenAICompatibleError` expanded from 4 → 6 stable categories:
- `.authFailed` (HTTP 401 / 403) — split from old `.badStatus`
- `.modelNotFound` (HTTP 404) — split from old `.badStatus`
- `.timeout` (`URLError.timedOut`) — split from old `.transport`
- `.networkFailed` (5xx, other 4xx, DNS, TCP, TLS, other URLError) — catch-all
- `.missingAPIKey` and `.malformedResponse` kept (already correct)

Every description prefixed with stable `[OPENAI_*]` tag for cross-language parsing. `category` property returns stable strings for any consumer that prefers that surface.

**Phase B (dreamforge, `5e8cb57`)**: New `src/lib/dreamProviderError.ts` parser + `DreamPanel.tsx` structured error view. Replaces raw `Error: ${err.message}` with `ProviderErrorView` that shows only the provider-controlled short message + fix-action button:
- missing-key / auth-failed / model-not-found → "Open Settings → AI" button (sets `settingsInitialSectionId = 'settings-section-ai'`, opens Settings)
- timeout / malformed / network-failed → "Retry" button (re-runs `dreamvault_run` with same args)
- unknown → "Dismiss" (no action)

**Security invariant locked**: test `never leaks API key value to the UI` injects `sk-or-v1-SECRET-LEAK-12345` as the body of a `[OPENAI_NETWORK_FAILED]` error and asserts the secret does NOT appear anywhere in the rendered DOM. Test `shortMessage is provider-controlled constant, not body-derived` pins that `shortMessage` and `fixActionLabel` are fixed constants from `PROVIDER_ERROR_MAP` — never derived from stderr content.

### PR 35 — Settings AI 简化 (commit `c9662a3`)

v0.5.0 `AiProviderSettings` rendered 5 fields in a 2-column grid (Provider / Name / Base URL / Model ID / API key storage + key/env inputs). New users saw a wall of fields with no clear flow, and the "API key storage" mode Select (local_file / env / none) was a 3-way choice most users don't need to think about.

**Restructured to 4-step main flow** (always visible):
1. Provider kind (Select)
2. Base URL (Input, pre-filled from kind)
3. Model ID (Input, pre-filled from kind's default)
4. API key (Password input, top-level — was nested under storage-mode Select before)

**Advanced** (collapsed by default via `<details>`):
- Custom name (defaults to catalog name; rarely overridden)
- API key storage mode (Select: local_file / env / none)
- API key env var (only shown when storage mode = "env")

`canSave` relaxed: name + baseUrl now optional (catalog defaults), modelId required, apiKey required when storage mode = `local_file` (default).

**i18n parity**: 1 new key `settings.aiProviders.advanced` = "Advanced" added to all 20 locales (English placeholder in non-English locales, ready for human translation). `i18n.test.ts` parity guard passes.

### PR 36 — Anthropic + Gemini adapter (commits `92352f2` Swift + `dd97878` dreamforge)

The remaining 2 cloud LLM providers per the v0.6 plan. Both deliberately split from OpenAI-compat (per user 2026-06-19 拍板: "provider 切分按协议相似度, 不同协议 推 v0.6 单独 PR, 否则你会同时改 4 个 surface 回归成本太高").

**Phase A (DreamVault Swift, `92352f2`)**:

**`AnthropicProvider`** — Anthropic uses its own Messages API, NOT OpenAI-compat:
- Endpoint: `POST {baseURL}/v1/messages`
- Auth: `x-api-key: {apiKey}` header (NOT `Authorization: Bearer`)
- Required header: `anthropic-version: 2023-06-01` (pinned)
- Body shape: `{ model, max_tokens, system, messages: [{role: user, content}] }` — system is a top-level string field, NOT a messages array entry. `max_tokens` is required (default 1024, enough for Consolidator short replies).
- Response: parses `content[0].text` (skips `tool_use` blocks; tool_use-only responses surface as `.malformedResponse`)

**`GeminiProvider`** — Gemini uses generateContent API, NOT OpenAI-compat, NOT Anthropic:
- Endpoint: `POST {baseURL}/v1beta/models/{model}:generateContent`
- The `:generateContent` is a literal path suffix (NOT a directory); preserved by manual string concatenation (URLComponents would percent-encode the colon to `%3A` which the API rejects)
- Auth: `x-goog-api-key: {apiKey}` header (newer; query param `?key=` is deprecated by Google)
- Body shape: `{ systemInstruction: { parts: [{text}] }, contents: [{role: user, parts: [{text}]}] }` — systemInstruction is top-level, contents has only the user turn
- Response: parses first text part of `candidates[0].content.parts` (skips functionCall parts; functionCall-only responses surface as `.malformedResponse`)

Both new providers:
- Use the same env-var boundary lock as OpenAI: `DREAMFORGE_LLM_API_KEY` injected by DreamX Rust PR 27, never read from macOS Keychain
- Throw the same 6-category error contract with provider-specific tag prefix (`[ANTHROPIC_*]` / `[GEMINI_*]`)
- `category` property values are provider-agnostic (just `missing-api-key` / `auth-failed` / etc.) — DreamX UI maps all 3 providers to the same short actionable copy because the user-facing fix is the same

**Cross-cutting**:
- `ResolvedLLM.Provider` enum: 3 → 5 cases (added `.anthropic`, `.gemini`)
- `parseProvider()`: accepts `"anthropic" / "claude"` (alias) / `"gemini" / "google" / "google-gemini"` (aliases) — colloquial user inputs
- `runtimeContext()` cloud-consent gate now covers all 3 cloud providers via new `requiresCloudConsent(_:)` helper (one-line change to add a 4th cloud provider in the future)
- `providerName(_:)` helper for budget / logging (Anthropic → "anthropic", Gemini → "gemini")

**Phase B (dreamforge, `dd97878`)**:
- `dreamProviderError.ts` parser recognizes 12 new tags (6 × 2 providers). Total: 18 tags, all mapping to the same 6 categories
- Cross-provider invariant test: same category across providers → same `shortMessage` / `fixActionLabel`. No provider name (openai / anthropic / gemini) leaks into the UI copy

## Closed-loop data flow (verified by unit tests, NOT by real provider call)

```
DreamX Settings → AI → Anthropic/Gemini provider config
    └─→ saveAiModelProviderApiKey (Keychain, PR 25 — unchanged)
          └─→ writeActiveLlmApiKeyEnv + writeActiveLlmApiKeyProviderId (localStorage, PR 26/27 — unchanged)
                └─→ DreamPanel reads both on "Run Dream" click (PR 27 — unchanged)
                      └─→ dreamvault_run { llmApiKeyProviderId, llmApiKeyEnv, ... } (PR 27 — unchanged)
                            └─→ --llm anthropic / --llm gemini flag (PR 30 — extended to new providers)
                                  └─→ Command::env("DREAMFORGE_LLM_API_KEY", value) (PR 27 — unchanged)
                                        └─→ dream subprocess env
                                              └─→ AnthropicProvider.init / GeminiProvider.init
                                                    ├─→ POST {baseURL}/v1/messages + x-api-key (Anthropic)
                                                    └─→ POST {baseURL}/v1beta/models/{model}:generateContent + x-goog-api-key (Gemini)
                                                          └─→ response parsed → Consolidator → Persister → MEMORY.md
```

**This data flow is identical to v0.5.0's verified path** for OpenAI-compat. The only thing v0.6.0 changes is the wire format and the request/response parser. The closed-loop invariant ("security boundary test ≠ end-to-end test" from §40) is preserved by the same testing approach: each layer (Keychain, env wiring, dream CLI arg, Swift provider, HTTP shape, response parser) has unit tests. The cross-layer invariant (env var contains the right value) is verified end-to-end for the OpenAI-compat case in v0.5.0 PR 32 and not re-verified here — adding new providers to an already-verified chain is a code-merge exercise, not a re-architecture.

## Post-ship addendum — 2026-06-22

After the v0.6.0 tag (`4989fb3`), the following fixes and verifications were completed. These do NOT change the v0.6.0 ship conclusion (which remains "API-complete and test-verified, real E2E deferred to v0.6.x") — they are post-tag improvements that strengthen the v0.6 → v0.6.x bridge and lock in learnings from running real cloud calls.

### Fixes / improvements

- **PR 37a (commit `8158d0a`)** — AI Settings copy polish
  - Catalog `name`: "Custom provider" → "Custom" + new i18n key `kind.custom` (all 20 locales)
  - "local key" / "本地密钥" / "Save locally in DreamX" → "macOS Keychain" / "macOS 钥匙串" in all 20 locales
  - "Advanced" → "高级设置" in zh-CN (the only locale with English residue)
  - Regression guards: 4 i18n tests + 2 aiTargets tests (catalog name + 3 cloud provider base URLs pinned)

- **PR 37b (commit `f7e1722`)** — wire `--llm gemini / --llm anthropic` based on provider kind
  - Root cause of "v0.6 Swift providers never get called from GUI" gap: dreamforge Rust `--llm` flag injection only handled `kind: "open_ai_compatible"`. For `kind: "anthropic"` and `kind: "gemini"`, no flag was injected, so dream CLI defaulted to MockLLMProvider.
  - Fix: `dream_llm_flag_for_provider_kind(kind)` helper + new `llmProviderKind` arg flowing Settings → localStorage → dreamvault_run → dream CLI flag. Accepts `"anthropic" / "claude"` (alias) / `"gemini" / "google" / "google_gemini"` (aliases).
  - 8 files, 758 Rust tests (+2), 3880+ TS tests (+5+). Provider-agnostic at Rust boundary — adding a 4th cloud provider is one line.

- **SettingsPanel auto-save (commit `6752c3a`)** — fix "Custom provider 关闭后丢失"
  - Symptom: open Settings → AI → fill in provider → click "Add provider" → close Settings without clicking the panel Save button → new provider gone.
  - Root cause: `AiProviderSettings.onChange` updated React draft state but did not call `onSave`. The just-added provider only persisted when the user hit the bottom Save button.
  - Fix: `handleAiModelProvidersChange` in SettingsPanel calls `onSave(buildSettingsFromDraft(...))` immediately when a provider is added, so it's in `settings.aiModelProviders` (persisted to settings.json) right away. Uses `draftRef` to keep callback identity stable across renders.
  - Test: "auto-saves added API model providers so closing settings does not lose them" locks the regression.

- **Dream CLI URL gotcha doc (DreamVault commit `606df99`)** — `--base-url` help explains direct CLI vs GUI behavior
  - Symptom hit during PR 37b re-verify: passing `--base-url https://api.siliconflow.cn/v1` to dream CLI directly got `[OPENAI_MODEL_NOT_FOUND] HTTP 404`. Switched to `https://api.siliconflow.cn` (no `/v1`) made the call succeed.
  - Root cause: dream CLI's Swift OpenAICompatibleProvider appends `/v1/chat/completions` to baseURL itself (no auto-strip). dreamforge Rust strips `/v1` for GUI users (PR 10), so GUI never hits this — only manual `dream run --base-url ...` does.
  - Fix: docstring-only. `dream help` now shows clear ✓ / ✗ examples and explains the GUI vs direct-CLI behavior.
